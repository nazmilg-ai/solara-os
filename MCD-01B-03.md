## Documentation Conventions: Advisor Selection / Derived Default / Supplier Capability

Consistent with MCD-01B-02's Documentation Conventions section, the following three terms are used throughout this document as **documentation conventions only** — they are not new architectural layers and do not modify MCD-01A §3:

- **Advisor Selection** — a value the advisor actively chooses. Architecturally, this is the Configuration layer (or, where the choice changes construction, the Construction/System layer).
- **Derived Default** — a value the system determines automatically from an Advisor Selection, without a separate question. Architecturally, this is the Accessory Engine and/or Supplier Capability Engine resolving a compatible component or value from an existing Configuration selection.
- **Supplier Capability** — the actual data (dimensions, weights, compatibility, pricing, availability) that makes a Derived Default resolvable. Architecturally, this is the Supplier Capability Engine (MCD-01A §5.9) and the Supplier Capability Library, maintained in MCD-04.

## 3. Construction Classification (required analysis — see Decision Register #10)

MCD-01A §3 defines a three-test model (Construction/System test → Configuration test → Accessory test, applied in order) for classifying any candidate concept. This section applies that test to each of the brief's four candidate constructions: Standard, Breakaway, No-Drill, Cassette. The full reasoning is recorded in Decision Register #10; this section states the conclusion each construction rule (§4) is built on.

**Standard Headrail** — passes test 1 (Construction/System test): it is the fundamental, baseline blind-building route (headrail mechanism, bracket type, assembly) against which the other constructions are compared. Construction/System.

**No-Drill Headrail** — passes test 1. It changes the fixing/mounting mechanism entirely (tension or spring-fit fixing into a recess rather than screw-fixed brackets), which is a bracket/assembly change in the sense MCD-01A §3 test 1 requires. It also introduces eligibility constraints (firm-backed recess surfaces only; blocked for hollow plasterboard) and distinct validation rules (minimum recess depth, clearance, manufacturer deductions) not applicable to Standard. Construction/System.

**Cassette Headrail** — passes test 1. This directly parallels MCD-01A §3's own canonical worked example for Roller Blinds ("Cassette fails test 1 to pass — it changes the headrail, bracket design, assembly, appearance, and sometimes the fabric deduction and tube selection — Construction/System"). Roman's Cassette Headrail is the same concept: a covered, fully enclosed headrail with different bracket design, appearance, and front-clearance requirements than Standard. Construction/System.

**Breakaway Headrail — reclassified.** This candidate fails test 1. A breakaway mechanism is a child-safety control device (typically a release point in the chain/cord loop, or a bracket detail designed to detach under excessive load) — it does not change the underlying headrail mechanism, bracket design, assembly route, or appearance the way Cassette or No-Drill do; the same Standard (or Cassette) headrail construction underlies it. This is corroborated architecturally: MCD-01A §5.6 already names "Breakaway devices" as a Child Safety Engine responsibility — a platform-wide concern applicable to any product with a chain/cord, not a construction-defining concept specific to Roman. It is also corroborated by this document's own Pricing Inputs (§18), which lists "construction uplift" and "breakaway mechanism" as separate, parallel pricing inputs — consistent with breakaway being an additive mechanism cost, not itself a construction.

Breakaway passes test 2 (Configuration test): for a Manual, chain/cord-operated Roman Blind, some compliant child-safety control mechanism must be selected — this is a mandatory selection in the sense test 2 requires, without changing the product's underlying construction or identity. Breakaway Control is therefore Configuration — a mandatory-category safety-mechanism selection, available only when Operation = Manual, alongside whichever other compliant safety-device options Supplier Capability offers. The physical breakaway device itself is an Accessory Engine item (§16), matching the precedent MCD-01B-01 §11 and MCD-01B-02 §14 both established for Bracket Type: the selection is Configuration, the physical part is an Accessory Engine item whose eligibility the selection and Construction Variant jointly determine.

**Conclusion:** §1, §4, and §17 are restructured accordingly. Standard, No-Drill, and Cassette are the three Construction Variants. Breakaway Control is a Configuration-layer, Manual-operation-only safety mechanism, not a fourth construction.

## 4. Construction Rules

### 4.1 Standard Headrail

Supports: Manual; Motorised, subject to Supplier Capability; Cascade or Stacked; Recess or Exact; Top fix or face fix; Breakaway Control, subject to Supplier Capability confirmation of eligibility. Motorised Standard Headrail has one compatible motor family per supplier construction — the advisor does not choose between unrelated motor models.

### 4.2 No-Drill Headrail

Supports: Recess measurement only; Manual operation unless separately confirmed; Firm-backed recess surfaces only; Maximum finished-weight validation; Construction-specific minimum recess depth and clearance; Manufacturer deductions applied from recess measurements. No-Drill must be blocked for: hollow plasterboard; surfaces without firm structural backing; unsuitable recess geometry; weights or dimensions exceeding Supplier Capability.

**Not yet specified:** whether Breakaway Control is offered on No-Drill — needs Supplier Capability confirmation (see Open Evidence Items, §26). Not assumed either way.

### 4.3 Cassette Headrail

Supports: Manual; Motorised; Covered or uncovered cassette where supported; Cascade or Stacked; Recess or Exact; Top fix or face fix; Breakaway Control, subject to Supplier Capability confirmation of eligibility. Cassette dimensions and front-clearance requirements are validated through Supplier Capability.

## 8. Measurement Rules

Measurement behaviour is provided by the shared Measurement Engine (MCD-01A §5.2) — its core responsibilities (Width, Height, Recess, Exact fit, Bay windows, Tolerances, Validation) are not restated here.

Roman supports Recess and Exact:
- **Recess** — the advisor records the recess size; supplier/manufacturing deductions are applied through Supplier Capability. This Product Specification does not hard-code one universal deduction.
- **Exact** — the advisor records the required finished blind size.

Survey guidance retains: measure width at top, middle and bottom; record the smallest relevant recess width; measure drop at left, centre and right; record the smallest relevant recess drop; identify handles, tiles, radiators, sills and other obstructions; verify front clearance against the selected construction.

Outside-recess recommendation guidance may include lateral overlap, top overlap, and bottom termination around sills and radiators. Supplier-specific overlap figures belong in MCD-04 or survey guidance records, not here.

## 9. Validation Rules (see Decision Register #12 for the Configured-vs-Extended reasoning)

The shared Validation Engine performs validation. Roman-specific validation includes, checked independently:
- Minimum width
- Maximum width
- Minimum drop
- Maximum drop
- Maximum total area (Area = Width × Drop; a blind can fail area validation even when width and drop each pass independently — see below)
- Maximum finished weight
- Fabric no-join width
- Construction eligibility and clearance
- Operation limits
- Motor minimum width
- Supplier live-ordering limits
- Same Room fabric/colourway mismatch (warn or block — see §14)

The system does not assume the first price-table cell is the minimum manufacturable size. Price tables may begin at a larger chargeable bracket than the technical minimum.

Where a supplier documents a maximum-area rule, the system calculates Area = Width × Drop and validates against it. Maximum-area values belong to Supplier Capability records (MCD-04), not a universal Roman rule, unless later adopted as a Solara-wide business policy (Decision Register #8).

**Confirmed via MCD-04 (reconciled 14 Jul 2026):** Standard Headrail's and Breakaway configuration's finished-width and finished-drop ranges, and No-Drill's maximum finished-weight limit, are now Supplier Capability facts — see MCD-04 for the specific values, not duplicated here per this document's supplier-neutrality rule.

**Not yet specified — needs supplier data before technical design:** No-Drill's and Cassette's own finished-width and finished-drop ranges (Standard's and Breakaway's are resolved — see above), and fabric no-join widths for individual fabric collections. Flagged here rather than invented (Open Evidence Items, §25).

## 10. Material, Fabric Property, and Lining Model

Fabric property and lining are separate configuration fields (Decision Register #2) — a blockout face fabric and a blockout lining are never treated as the same selection.

Fabric properties may include: Sheer, Dimout, Blockout face fabric, Standard furnishing fabric.

Lining options may include, according to Supplier Capability: Unlined, Standard lining, Premier lining, Satin lining, Blockout lining, Bonded lining, Bonded blockout lining, Interlining.

The system filters lining and stitch options according to the selected fabric and Supplier Capability.

## 11. Fabric Join Rule

Each fabric record (MCD-04) includes: usable fabric width; maximum blind width without joins; whether joined manufacture is permitted; maximum joined width; join arrangement where known.

As soon as the selected blind width exceeds the selected fabric's usable no-join width, the system displays:

```text
Fabric joins required
This Roman Blind exceeds the selected fabric's usable width and will be manufactured with visible vertical joins. Please explain this to the customer before proceeding.
```

The advisor records: "Customer informed that visible fabric joins will be present." This warning occurs during configuration, before quotation acceptance, and before supplier-order submission if not already acknowledged (Decision Register #3).

Where possible, the Recommendation Engine may suggest a wide-width fabric that avoids joins — it advises, it does not force (see MCD-01A §5.10's governing principle: the advisor retains complete control).

## 13. Motor Implementation

### 13.1 General rule

Motorisation is construction-dependent. The advisor selects: Manual or Motorised; compatible construction; control requirements. The Supplier Capability Engine determines: compatible motor family; minimum motor width; maximum dimensions; power arrangement; suitable control accessories; load suitability. Supplier-specific motor names do not appear in this document.

### 13.2 Standard Headrail

For a motorised Standard Headrail: one compatible motor family is automatically assigned (Derived Default); no advisor motor-model choice is presented; if the configuration exceeds the motorised construction limit, the system blocks or refers.

### 13.3 Cassette Headrail

Cassette motorisation may have lower-capacity and higher-capacity motors. **No specific dimensional threshold (e.g. a width or drop figure in mm) is encoded as an operative rule.** An earlier draft of this brief included a provisional 1500mm boundary; that figure originated from advisor risk-management judgement, not supplier evidence, and has been removed rather than merely labelled provisional (Decision Register #7).

Where motor-capacity selection between lower- and higher-capacity options cannot yet be determined from Supplier Capability data, the system defers entirely:

```text
Motor selection subject to Supplier Capability confirmation.
```

The system never downgrades a configuration already confirmed to require a higher-capacity motor once evidence establishes that requirement.

### 13.4 No-Drill Headrail

Defaults to Manual per §4.2. No dedicated motor family is assigned unless a future Supplier Capability record confirms motorised No-Drill eligibility (not assumed either way — see Open Evidence Items, §26).

### 13.5 Load model

The architecture supports future calculated-load validation using: estimated lifted load = face fabric weight + lining weight + interlining or bonded-layer weight + rods + bottom bar + cord and pocket allowance + joined-panel allowance.

No fabric weights or motor-load thresholds are invented. Where evidence is missing: "Motor selection subject to Supplier Capability confirmation."

## 14. Same Room Option (see Decision Register #11 for the required ownership analysis)

The Same Room Option links separate Roman Blind items. When enabled, the advisor either adds the item to an existing Same Room Group or creates a new one. All grouped items must use the same fabric, the same colourway, compatible construction, compatible lining and stitch method where required, and the same supplier pattern-matching instruction. The system warns or blocks grouping where fabric or colourway differs (Validation Engine, per §9 and Decision Register #12).

Supplier order instruction:

```text
Pattern-match all Roman Blind items in the same Same Room Group using the same fabric and colourway. Coordinate the pattern consistently across the grouped items, subject to supplier manufacturing tolerances.
```

Same Room Option is distinct from pattern matching within a single blind (§15). Supplier-specific tolerances, such as permitted drop variation for fold alignment, are retained in MCD-04, not here.

**Architectural ownership (see Decision Register #11 for the full reasoning):** cross-item grouping and deviation detection is attributed to the Survey Engine, whose MCD-01A §5.1 responsibilities already include creating multiple products within a single quotation, duplicating previously entered configurations, detecting differences between duplicated items, highlighting deviations, requiring advisor confirmation where deviations exist, and maintaining a complete audit trail — the same shape of behaviour Same Room Option requires, applied to a persistent named group rather than a one-off duplicate-and-compare action. This is not treated as a genuine architectural gap; it is not assigned to Audit & Compliance Engine by default, notwithstanding the brief's provisional table placement.

## 17. Child Safety

The shared Child Safety Engine implements BS EN 13120 compliance (MCD-01A §5.6). Manual looped-chain Roman Blind systems require: installation height; chain-length calculation (drop-derived, a Child Safety Engine calculation, not a Configuration input, consistent with MCD-01A §3's explicit chain-length note); P-clip or equivalent safety device; Breakaway Control where selected (Configuration-layer, per §3 — not tied to a "Breakaway Headrail" construction, since no such construction exists in this document); rear lift-cord breakaway clips.

Motorised operation removes the manual chain question entirely.

Because Breakaway Control is a Configuration-layer alternative among the available compliant safety-device options (not a separate construction), the survey does not present incompatible or redundant safety-device questions once a safety mechanism has been selected — e.g. a fixed P-clip mapping and Breakaway Control are alternative choices for the same underlying requirement (some compliant safety device must be selected for Manual chain/cord operation), not both required simultaneously. When Breakaway Control is selected, the system filters out fixed-loop chain/P-clip safety questions, since Breakaway Control uses a different safety mechanism and those questions do not apply to it.

The exact chain formula and safety device mapping come from Supplier Capability and the Child Safety Engine (Open Evidence Items, §26).
